### Shortlist Ideas (Pre-Evaluation)

**Idea 1: MV-Refresh Optimizer for PostgreSQL**
- Problem: Materialized views on large tables (analytics.classification 95M rows) timing out during CONCURRENTLY refresh. Existing solution: removed CONCURRENTLY + increased timeout to 10min. But as data grows, this will timeout again. Partitioning mentioned as Option 3 but not implemented.
- Solution: Automated partitioning + incremental refresh strategy. Create monthly partitions on flattened_interaction, only refresh recent partitions. Maintain partition manager that auto-creates new partitions based on data. Refresh orchestration that runs in parallel with timeout management.
- Simplicity: Uses native PostgreSQL partitioning, existing Temporal infrastructure. Could be implemented as a helper library + cron/Temporal jobs.
- Maintenance: Once set up, self-managing partitions. Would need monitoring for partition creation failures.
- Bloat Signals: No new language, pure SQL/TypeScript. Leverages existing DB infra.
- Utility Score: 8/10 (solves a critical scaling bottleneck that will worsen over time)

**Idea 2: Classifier Evaluation Framework**
- Problem: Classifier system uses case statements that are "hectic" and hard to maintain. Need evals system to measure accuracy and guide improvements. Manual audit with spreadsheets mentioned repeatedly. Current approach: run backfills, look at Hex dash, manually listen to calls. No systematic evals pipeline.
- Solution: Build a labeled dataset management system with:
  - Audio snippet storage (GCS) linked to interaction IDs
  - Human labeling interface (simple web UI or integrate with existing tools)
  - Automated evaluation runs comparing classifier outputs vs. ground truth
  - Metrics dashboard (precision/recall per classifier, confusion matrices)
  - Versioning of classifier prompts/configs
- Simplicity: Could start minimal—just a table + simple UI. Use existing GCS storage. Don't over-engineer.
- Maintenance: Would need periodic human labeling (QA team). Framework itself could be low-maintenance once stable.
- Bloat Signals: Introducing a new web UI adds complexity. But could be as simple as a Next.js app (existing tech stack) or even Hex integration.
- Utility Score: 9/10 (addresses a fundamental gap in the analytics pipeline; quality improvements would impact all tenants)

**Idea 3: Ingest→Event Scheduler Bridge**
- Problem: Moving from Inngest to Temporal. Need to replicate the "populate interactions_to_refresh" pattern where table changes trigger refresh workflows. Current Temporal migration is piecemeal (task-and-classification first). Need a generalized system.
- Solution: Build a lightweight change data capture (CDC) bridge that watches key tables (salesforce sync tables, inbound_phone_call, customer_interaction, etc.) and writes to an "events" table. Temporal workers listen to that table and spawn appropriate refresh workflows. Could use PostgreSQL LISTEN/NOTIFY or polling with cursor.
- Simplicity: Minimal—just a few tables and workers. No new frameworks.
- Maintenance: Need to monitor event backlog, ensure no dropped events. But Temporal workers naturally handle retries.
- Bloat Signals: Risk of over-engineering. Could start with specific triggers rather than a generic system.
- Utility Score: 7/10 (facilitates the migration but may be temporary if a full event-sourcing architecture emerges later)
