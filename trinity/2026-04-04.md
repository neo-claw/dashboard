## 23:30

### Thought
- Recent context: Yosemite campground monitoring script exists, but has a mapping bug (Tuolumne Meadows instead of North Pines). Notifications currently go to Twitter, but the user's OpenClaw runtime uses Telegram. There's opportunity to improve integration and add intelligence (cancellation detection, health monitoring).
- Goal: Build utility-focused tools for Neo with minimal bloat.

### Exploration & Ideas
1. **Telegram Notifier Integration**
   - Problem Fit: High. Notifications should land where user actually is (Telegram). Twitter integration is misaligned.
   - Simplicity: Medium. Requires Telegram Bot token and chat_id, but simple HTTP POST. No new frameworks.
   - Maintenance Cost: Low. Only credentials and endpoint. Stable Telegram API.
   - Bloat Signals: None. Straightforward replacement, minimal code.
   - Utility Score: 9/10

2. **Cancellation Detection (Stateful Diff)**
   - Problem Fit: High. Users want to know when sites free up; current script alerts on every run if any sites available (may be noise). Detecting changes (newly available) is more valuable.
   - Simplicity: Medium. Need to persist previous availability JSON and compute diffs. Small persistence logic.
   - Maintenance Cost: Low. No external deps beyond file I/O.
   - Bloat Signals: None. Pure logic addition.
   - Utility Score: 9/10

3. **Cron Health Monitor**
   - Problem Fit: Medium-High. Cron jobs can silently fail; a watchdog that checks timestamps and output health would catch issues.
   - Simplicity: Medium. A small wrapper script that checks last run time, exit code, and output patterns. Could use existing heartbeat/monitoring.
   - Maintenance Cost: Low. Minimal logic.
   - Bloat Signals: None. Lightweight.
   - Utility Score: 8/10

4. **Config Fix (North Pines)**
   - Problem Fit: High but trivial. Correcting campground IDs is necessary but not a "build".
   - Simplicity: Very High. Just edit a config.
   - Maintenance Cost: None.
   - Bloat Signals: N/A.
   - Utility Score: 5/10 (important but not a new tool; will do regardless).

### Decision
- Highest scores: A and B both 9/10.
- I will combine them into a single utility: **A Telegram-native, stateful notifier** that:
  - Sends alerts via Telegram instead of Twitter.
  - Persists previous availability and only notifies when new sites appear (cancellation detection).
  - This builds on existing code, reduces bloat by removing Twitter dependency, and adds intelligent filtering.
- Expected outcome: a new notifier module that can be dropped into the existing pipeline.
